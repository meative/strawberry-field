00いま、どこにいるかWhere We Are

4つの画面（サロン予約・予約ボード・お迎え通知・受付アプリ）が完成し、同じパソコンの中なら本物のように連動して動く状態です。これはクラウド化の「足がかり」が整った段階です。これから進む全体の流れは次の3つのフェーズです。

まず集中するのは Phase 1・2。Phase 3 はその先の本命です。

01各部署に配ってテストしてもらうPhase 1 ・ Testing

各部署に渡すのは、完成した4つのHTMLファイルそのものです。メール・共有フォルダ・USBなどで渡し、各端末のブラウザで開いてもらいます。

⚠ テスト段階での最重要ポイント

画面どうしの連動（予約が他の画面に反映される動き）を確認するには、4つの画面を「同じ1台の端末・同じブラウザ」で開く必要があります。今はまだ各画面が同じパソコンの中だけでデータを共有する仕組みのため、別々のパソコンに分けて配ると連動しません。テスト中は「サロン役・保育園役を1台で切り替えて触る」形が確実です。
※ 拠点をまたいだ連動は、Phase 3 のクラウド化で実現します。

02フィードバックを集めて修正を重ねるPhase 2 ・ Refine

テストで出た「使いにくい」「こうしたい」という声を集め、画面を直していくサイクルです。修正版を作って配り直す、を繰り返します。

声は「具体的に」集めるのがコツ

次の4点がわかると、修正がすぐにできます。

  • どの画面か（例：サロン予約画面）
  • どこで（例：時間を選ぶところ）
  • 何をしようとして（例：30分単位で予約したかった）
  • どうなってほしかったか（例：もっと細かく選べると嬉しい）
＜修正サイクル＞ 小さく回して、確実に良くする

03整ったらクラウド化へPhase 3 ・ Cloud

テストが落ち着き、画面の作りが安定したら、いよいよ本命のクラウド化（Firebase）です。これで拠点をまたいで予約・通知がリアルタイムに連動し、全部署で共有して使えるようになります。詳しい手順は別資料「Firebase化 ロードマップ」にまとめています。

クラウド化の前に、関係者と決めておくこと

これらはテスト期間中に並行して話し合っておくと、クラウド化がスムーズです。

  • 個人情報の扱い — お子様・保護者の情報をクラウドに置くため、誰がどのデータを見られるか（権限）のルール
  • 各拠点のネット環境 — 保育園・ウッドストック・ジャニスに安定したWi-Fi等があるか
  • 運用ルール — 予約の変更・キャンセルの扱い、余裕活用型の枠配分など
