# STRAWBERRY FIELD 保育予約・お迎え通知システム ─ プロジェクト概要書

> 新規プロジェクトのナレッジに入れて使う引き継ぎ資料です。概要・データ構造・連携の法則（仕様）・運用手順・開発ルールをまとめています。

---

## 1. プロジェクトの全体像

札幌の保育園グループ「STRAWBERRY FIELD」（4園）の業務システムのプロトタイプです。系列の美容室「ウッドストック（WOODSTOCK）」・まつげサロン「ジャニス（JANIS）」を利用するお客様が、施術中に隣接する保育園へお子様を無料で預けられる仕組みを、3つの画面で支えます。

現状は **3つのHTMLファイルが localStorage を介して連携するプロトタイプ** です。実運用に向けた「足がかり」であり、最終的にはサーバ（Firebase等）連携に置き換える前提で作っています（詳細は §8）。

### デザインテーマ「エレガント・ニューヨーク」
- Tiffanyブルー系 `#0ABAB5`（deep `#089A95`）、シャンパンゴールド `#C9A961`（deep `#A88A4A`）
- 書体：見出し英字 Cormorant Garamond（イタリック）、和文 Noto Serif JP
- サロン別カラー：WOODSTOCK = ゴールド系、JANIS = ブルー系 `#5B8DB8`
- 基準日（本日）は3画面すべて **2026年5月23日（土）** に統一

## 2. 3つの画面（ファイル）

| ファイル | 役割 | 使う人 |
|---|---|---|
| **サロン予約画面_v2.html** | サロン側で保育予約を入れる画面。年齢・月齢・時間を選んで空き状況を確認し予約。次回予約も登録可能。 | 美容室／まつげサロンのスタッフ |
| **予約管理システム_v2.html** | 保育園側の予約管理ボード（タイムライン形式）。全予約を時間軸で俯瞰。新規客の受付連携もここ。 | 保育園のスタッフ |
| **お迎え遅延通知_連携デモ_v2.html** | サロン↔保育園の双方向お迎え通知。施術の進捗・遅れ・お迎えを伝え合う。予約修正もここに集約。 | サロン・保育園 双方 |

3画面は独立したHTMLですが、同じブラウザの localStorage を共有して連携します。

## 3. 共有データ構造（最重要）

### 3-1. 予約データ：localStorageキー `sf_bookings_v1`
3画面が読み書きする予約の配列。1レコードの構造：

```
{
  id,                       // レコードID（一意）
  salon,                    // 'woodstock' | 'janis'
  name,                     // 表示名（新規は当初カタカナ名、受付後は正式氏名）
  kana,                     // ふりがな（カタカナ）。TimelySchool照合キー
  age,                      // '0' | '1' | '2' | '3'
  months,                   // 月齢 2〜11（3歳はnull）
  start, end,               // 'HH:MM' お預かり開始・終了
  date,                     // 'YYYY-MM-DD'
  nextReservation,          // {date,time,menu} | null（次回予約）
  fromNextOf,               // 親レコードのid（将来日付の次回予約レコードに付く）
  memo,                     // メモ（アレルギー等）
  provisional,              // true=新規の一時予約（受付待ち）
  info                      // 受付反映済みの詳細 | null
}
```

### 3-2. TimelySchool受付データ：localStorageキー `timely_school_data_v1`
別アプリ「TimelySchool 一時預かり受付」が保存するデータ。予約管理ボードが**読み取り専用で参照**して照合に使う。

```
{ customers:[...], visits:[...], reservations:[...], students:[...], settings:{...} }
```

`customers` の1件（照合に使う主な項目）：
```
{ id, childName, childKana, gender,
  birthY, birthM, birthD,        // 生年月日 → 月齢を自動算出
  allergyHas, allergyItems[], allergyDetail,
  parentName, parentKana, parentTel, emergency1Name, emergency1Tel, ... }
```

### 3-3. リアルタイム反映の仕組み
- `storage` イベント（別タブの変更を検知）＋ 4秒ごとのポーリングの二段構え
- どの画面で予約・変更しても、他の2画面に自動反映される

## 4. 機能仕様（法則）

### 4-1. 年齢と月齢
- 年齢タブは 0歳 / 1歳 / 2歳 / 3歳〜 の4つ
- 0・1・2歳を選ぶと **月齢チップ（2〜11ヶ月）** を表示。**お預かりは2ヶ月から**なので0・1ヶ月は無し。横6列×2行で表示
- 3歳以上は月齢なし（タブのみ）
- 表記は「0歳3ヶ月」「3歳〜」の形式で3画面共通

### 4-2. 受付時間と定員（年齢別）
| 年齢 | 受付時間 | 同時人数 |
|---|---|---|
| 0歳 | 09:00〜15:00 | 2名 |
| 1歳 | 08:00〜18:00 | 3名 |
| 2歳 | 08:00〜18:00 | 4名 |
| 3歳〜 | 09:00〜18:00 | 2名 |

予約時・修正時はこの受付時間を超えないようガードがかかる。

### 4-3. 登録済み客 と 新規客（一時予約）
- 予約モーダルの氏名欄は1つに統合
- **登録済みのお客様**：氏名を入力すると候補が表示され、選ぶと本予約
- **新規のお客様**：カタカナで氏名を入力すると「一時予約」（`provisional: true`）。予約表に「新規・受付待ち」バッジ＋破線模様で表示

### 4-4. TimelySchool連携（新規客の正式情報の自動反映）★重要な仕様
新規客の情報は、保育園側で手入力するのではなく **TimelySchoolの受付内容を自動で取り込む**。

```
[流れ]
① サロンで新規客をカタカナ名で一時予約（例：タカシマ ヒデノリ）
② 新規客が保育園に来園 → 一時預かり受付（TimelySchool）を記入
③ 予約管理ボードが timely_school_data_v1 を読み、ふりがな（カタカナ）で照合
④ 一致したら、正式氏名（高嶋 仁）・月齢（生年月日から自動算出）・
   アレルギー（卵・乳 等）を予約レコードに自動反映
⑤ provisional を解除し、サロン側・お迎え通知にも正式情報が反映
```

- **照合キー：ふりがな（カタカナ名）**。`childKana` ↔ 一時予約の `kana` を正規化して照合
- 月齢は生年月日と基準日（5/23）から算出
- かつての「保育園側でインフォメーションシートを手入力」は**廃止**

### 4-5. 次回予約
- サロン予約画面の上部「今回の予約／次回予約」タブで切替（同じ検索UIを使い回す）
- 次回予約を登録すると、(1) 本日の予約に `nextReservation` を紐付け、(2) 将来日付の予約レコード（`fromNextOf`付き）を作成
- お迎え通知画面・受信ボードに「🗓 次回予約もしています（日時・メニュー）」と表示
- お見送り（「もう向かっています」）時、保育園へ次回予約も併せて告知

### 4-6. 予約の修正（時間・日にち）
窓口は **お迎え通知画面に集約**。
- **来園前（予約済み）の子**：カードの「🕐 予約の時間・日にちを変更」から、カレンダーで日にち・開始時刻・長さを選び直せる。別日に移すと本日リストから外れ、移動先の日付に登録
- **次回予約の変更**：滞在中でも、「🗓 次回予約もしています」欄をタップして日時・メニューを変更可。加えて、予約管理ボードでカレンダーから将来日付に移動し予約バーをタップしても変更可（両経路）
- お預かり中・お迎え対応中・完了の子は日にち変更不可（運用が壊れるため）

### 4-7. お迎え・遅延通知（双方向）
- サロン → 保育園：「あと10〜15分」「まもなく終わりそう（早め）」「もう向かっています」「遅れます（+10分〜）」「早く終わる（−10分〜）」
- 電話確認フロー：来園前に電話 → 来園時刻・お戻り時刻を連動更新
- 未到着の自動アラート（受付時間経過で no-show）
- 保育園 → サロン：「お子様の様子」を伝えるレポート（ごきげん／お昼寝 等）
- 受信ボードは受付に常時表示しておく通知ボードとして機能

### 4-8. カレンダー（自作）
- ネイティブの日付入力は使わず、月表示の自作カレンダー（サロン予約画面・予約管理ボード・お迎え通知の修正に搭載）
- 月送り（年またぎ対応）、日曜赤・土曜ゴールド、本日枠囲み、選択塗りつぶし、「本日に戻る」ボタン
- 予約管理ボードのカレンダーは、予約がある日付にゴールドのドットを表示（反映確認用）

## 5. 運用手順（デモの触り方）

### 5-1. 準備
1. 3ファイル（＋ TimelySchool受付）を**すべてダウンロード**
2. **同じブラウザ**で各HTMLを開く（別タブ可）
   - ※ Claudeのプレビュー内では localStorage 連携は動きません。必ずダウンロードして開くこと

### 5-2. 基本シナリオ
1. **サロン予約画面**で年齢・月齢・時間を選び、登録済み客を予約 → 予約管理ボード・お迎え通知に反映
2. **新規客**はカタカナ名で一時予約 → 予約管理ボードに「新規・受付待ち」で表示
3. **TimelySchool**で同じふりがなの子を受付記入 → 予約管理ボードに正式氏名・月齢・アレルギーが自動反映
4. **お迎え通知画面**で「預かり完了」→ 施術進捗や遅れを送信 → 受信ボードに反映
5. **次回予約**を登録 → カレンダーでその日付に移動して反映を確認 → 必要なら次回予約欄やボードから日時変更

## 6. 開発ルール（次の担当者・次のセッション向け）

### 6-1. 検証は必ず JSDOM で行う
- ネットワーク制限で実ブラウザ（Chrome）が使えないため、各 `<script>` を抽出して Node の JSDOM 上で関数テストする
- 手順：`<script>` 抽出 → `api.anthropic` を含むartifact内スクリプトを除外して結合 → `node --check` で構文確認 → JSDOMで `vm.runInContext` → 全パス確認後に出力
- localStorageモックは「実際に読み書きする」形にする（`getItem:()=>null` の空モックだと種まきが効かずテストが失敗する）
- AudioContext / setInterval / requestAnimationFrame / alert もモックが必要

### 6-2. テストの「失敗」の多くはテスト側の問題
過去に頻出した誤検出：
1. 受付時間を超える時刻設定（ガードは正常動作）
2. localStorageモックの不備
3. 操作順序の誤り（例：次回予約は先にカレンダーで日付を設定してから confirm）
4. ネストしたテンプレートリテラルのバッククォート構文エラー

→ 失敗時はまず実コードでなくテスト側を疑う。ネストした `` ${...?`...`:`...`} `` はJS構文エラーになるので**文字列連結に書き換える**

### 6-3. データ・互換性のルール
- レコードの照合は **sfId（id）優先**、なければ 同名＋同開始時刻
- 取り込み済みレコードも、時刻変更・正式氏名化・月齢更新を**反映**する（スキップしない）
- 重複防止を常に意識（次回予約の将来レコードは作り直し時に旧レコードを削除）
- 基準日 `new Date(2026,4,23)` を3画面でずらさない

### 6-4. UI・トーンのルール
- 「エレガント・ニューヨーク」を維持（§1のカラー・書体）
- 角丸・ピル型・アイボリー帯トグルなど既存の意匠に合わせる
- 文言は丁寧でやわらかい接客トーン

## 7. 用語

| 用語 | 意味 |
|---|---|
| 一時予約 / provisional | 新規客がカタカナ名だけで入れた仮の予約。TimelySchool受付で正式情報に置き換わる |
| 次回予約 | 今回の施術中に取る、次回来店時の予約。本日レコードに紐付く＋将来日付のレコードも作る |
| TimelySchool | 別アプリの「一時預かり受付」。新規客の正式情報の入力元。`timely_school_data_v1` に保存 |
| 受信ボード | お迎え通知画面の保育園側。受付に常時表示する通知ボード |
| 基準日 | デモの「本日」。2026年5月23日（土）に固定 |

## 8. 残作業・今後の展望

- **本丸の Firebase 化**：現在の localStorage 連携を、サーバ経由のリアルタイム連携に置き換える。これにより別端末（サロンのPC・保育園のタブレット等）間でも枠が自動連動し、通知がリアルタイムに届く。今回のlocalStorage連携はその設計を先取りした足がかり
- 受付アプリ TimelySchool（v36）のスタッフテストのフィードバック反映
- 余裕活用型の年齢別枠配分ルールの確定（例：月寒園8名の配分）
- 関係者へのデモ共有
